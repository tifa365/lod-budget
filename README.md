# Berlin Budgetary Data

This repository is the home for the official budgetary data for Berlin by the Senate Department for Finance (Senatsverwaltung für Finanzen) as RDF / Linked Open Data.

**[Start browsing the data!](https://berlin.github.io/lod-budget/)**

Currently contained in the dataset are data regarding the years 2022–2027.

## Berlin's Budget Structure

The Berlin budget follows a strict three-level hierarchy: **Einzelplan → Kapitel → Titel**.

The top level, **Einzelpläne**, falls into three categories: constitutional bodies (01–02), Senate departments (03–15), and Berlin's 12 districts, each with their own Einzelplan (31–42) — reflecting Berlin's dual role as both city and federal state.

**Kapitel** are subdivisions within each Einzelplan, with a systematic numbering: ..01–09 for subject programs, ..10 for miscellaneous, ..11 for central administration.

**Titel** are the atomic budget line items, each with a 5-digit number and a functional classification code (Funktionskennziffer) indicating the area of government activity.

Each Titel is classified along two additional axes:

- **Gruppierungsplan** (economic classification): answers "what type of expense/income?" via Hauptgruppe → Obergruppe → Gruppe
- **Funktionenplan** (functional classification): answers "what government purpose?" via Hauptfunktion → Oberfunktion → Funktion

This triple classification enables cross-cutting queries such as "how much do all departments spend on construction?" or "what is the total education budget across all districts?".

### Berlin's Accounting System

Berlin uses **Erweiterte Kameralistik** — a hybrid accounting system unique among German federal states. Rather than switching to full double-entry bookkeeping (Doppik) like other states, Berlin extended its traditional cash-based system (Kameralistik) with cost accounting, performance metrics, and partial asset tracking. This achieves most of the transparency benefits of Doppik without the costly system migration.

The dataset contains only **Soll** (planned/budgeted) amounts. Actual spending (Ist) data is not currently included.

## Data Pipeline

The build pipeline merges multiple RDF sources into a single queryable knowledge graph:

| Source | Content |
|--------|---------|
| `data/haushalt-be.ttl` | Main budget data (covering 2022–2027) |
| `data/bezirke-be.ttl` | Geographic/administrative district data |
| `data/senfin_extra.ttl` | Additional Senate Finance metadata |
| `void.ttl` | Dataset description and statistics |
| SenFin organigram (fetched from GitHub) | Current organizational structure |

These are converted from Turtle to N-Triples and merged into `data/temp/all.nt`, which drives the static site generator.

## Generating the Site

Install dependencies:

```shell
uv sync
```

Build the merged dataset:

```shell
make data/temp/all.nt
```

Generate the static site:

```shell
make generate
```

Or generate and serve locally at `http://localhost:8000`:

```shell
make generate+serve_locally
```

## Updating the Budget Data

Source CSVs are published by SenFin on the Berlin Open Data portal:

| Years | Source |
|-------|--------|
| 2022/2023 | [daten.berlin.de](https://daten.berlin.de/datensaetze/doppelhaushalt-2022-2023-2-nachtrag-1464527) |
| 2024/2025 | [daten.berlin.de](https://daten.berlin.de/datensaetze/doppelhaushalt-2024-2025-2-nachtrag-1468058) |
| 2026/2027 | [berlin.de/sen/finanzen](https://www.berlin.de/sen/finanzen/service/daten/260223_doppelhaushalt_2026_2027.csv) |

When a new CSV is published, convert it, validate, and check the merged file:

```shell
# 1. Convert
uv run python bin/csv_to_rdf.py <input.csv> -o /tmp/new-years.ttl [-e utf-8-sig]

# 2. Validate row-for-row against source CSV (proves semantic correctness of data payload)
uv run python bin/validate_conversion.py <input.csv> /tmp/new-years.ttl [-e utf-8-sig]

# 3. Merge into the existing TTL, then check the combined dataset
uv run python bin/check_dataset.py data/haushalt-be.ttl
```

> **Note on IRIs:** `bin/validate_conversion.py` proves that every CSV row became the correct RDF observation with the correct amount. It does not guarantee IRI continuity with the original 2022–2025 file, which was produced by an external tool with an unknown UUID scheme. The 2022–2025 historic IRIs are preserved as-is; only new years use IRIs generated by this converter.

CSVs from 2026/2027 onwards use UTF-8 with BOM encoding (`-e utf-8-sig`) and zero-padded codes (e.g. `Einzelplan: "01"`). The converter normalizes these to match existing URI patterns.
